# 查找引用字段（Lookup）实现方案

> 目标：在当前 AI Filter for Lark Base 项目中，把已声明但未实现的 `Lookup` 字段类型落地，对齐飞书多维表格「查找引用」的语义——**无需依赖关联字段，直接按条件到任意目标表捞值并聚合**。

---

## 1. 背景与现状

项目中 `Lookup` 已存在类型骨架但无运行时语义：

| 位置 | 现状 |
|---|---|
| `backend/src/types.ts` | 已声明 `FieldType = "Lookup"`；有 `LookupConfig`、`LookupCondition`、`LookupCalcMethod` |
| `backend/src/services/fieldValidator.ts:136` | Lookup 归为只读字段，禁止写入 |
| `backend/src/services/filterEngine.ts:213` | 仅按文本/数字粗略比对 `raw`，未计算 lookup 实际值 |
| `backend/src/services/dataStore.ts:330` / `dbStore.ts:34` | 列入 "system field"，建表/导入时跳过单元格写入 |
| 前端 | `frontend/src/types.ts` 完全没有 Lookup 概念，UI 无配置面板 |

**本方案要做的事**：补齐「配置 → 存储 → 计算 → 展示 → 联动更新」全链路。

---

## 2. 语义定义（与飞书一致）

**查找引用字段 = 在当前表的每条记录上，按条件到目标表中筛选出一组记录，取其某字段值，再按所选计算方式聚合。**

形式化表达：

```
lookup(record_i) =
    calcMethod(
        ref_field 的值
        FROM  ref_table.records
        WHERE  匹配(conditions, record_i)        // AND / OR
    )
```

其中 `匹配(conditions, record_i)` 的每个条件可以是：
- `ref_table.字段 op 当前表.字段`（字段对字段）
- `ref_table.字段 op 常量`（字段对常量）

这与 Airtable 必须先建 Link、沿链跟过去的 Lookup 不同——**此处不需要预建关联字段**。

---

## 3. 数据模型

### 3.1 复用已有类型（不动）

```ts
// backend/src/types.ts 现状，沿用
export type LookupCalcMethod =
  | "original" | "deduplicate" | "deduplicateCount" | "count"
  | "sum" | "average" | "max" | "min";

export interface LookupCondition {
  refFieldId: string;               // 目标表中的字段
  operator: FilterOperator;         // 复用现有运算符集合
  value?: CellValue;                // valueType=constant 时使用
  valueType: "field" | "constant";
  currentFieldId?: string;          // valueType=field 时使用（新增，见 3.2）
}

export interface LookupConfig {
  refTableId: string;
  refFieldId: string;
  conditions: LookupCondition[];    // 上限 5
  conditionLogic: "and" | "or";
  calcMethod: LookupCalcMethod;
}
```

### 3.2 必要补丁

| 补丁 | 原因 |
|---|---|
| `LookupCondition` 增加 `currentFieldId?: string` | 原定义有 `valueType: "field"` 但缺少"当前表哪个字段"的引用 |
| `FieldConfig` 增加 `lookupOutputFormat?: string` 与 `lookupOutputType?` | 计算结果格式化（如 sum 后用货币还是数字） |
| `LookupConfig` 增加 `conditions.length <= 5` 的校验 | 与飞书一致 |

### 3.3 存储策略

**计算结果不持久化**：
- 当前 `Table.fields`/`records` 都在 JSONB 中；Record.cells 只存用户可写字段的值
- Lookup 字段视为**派生视图**，由 `lookupEngine` 在读路径（`viewEngine.queryView`）按需计算
- 理由：避免写放大、避免跨表触发器复杂度；项目规模小，内存/PG 计算即可

**可选优化**：命中率高时可在 Record 上增加 `derivedCells: Json`，用增量策略刷新，见 §7.2 阶段三。

## 4. 计算引擎（`backend/src/services/lookupEngine.ts`）

### 4.1 接口

```ts
// 单条记录求值
computeLookup(
  field: Field,                 // Lookup 字段定义
  record: TableRecord,          // 当前记录
  ctx: LookupContext            // 包含：当前表 fields、目标表 fields+records
): CellValue;

// 批量（供 viewEngine 使用，按 refTableId 预加载目标表）
computeLookupBatch(
  field: Field,
  records: TableRecord[],
  ctx: LookupContext
): Map<string /*recordId*/, CellValue>;
```

### 4.2 算法

```
输入: field.config.lookup = cfg, record_i, refTable
1. 解析目标表：refTable = tables[cfg.refTableId]
2. 筛选候选记录：
     matched = refTable.records.filter(r =>
       evalConditions(cfg.conditions, cfg.conditionLogic, r, record_i)
     )
3. 提取字段值数组：
     values = matched.map(r => r.cells[cfg.refFieldId])
     values = flatten(values)     // 多选/用户等数组型需展平
4. 按 calcMethod 聚合：
     original         → values（数组原值）
     deduplicate      → unique(values)
     count            → values.length
     deduplicateCount → unique(values).length
     sum / average / max / min → 仅对可 coerce 为 number 的值生效
5. 返回结果
```

**条件求值 `evalConditions`**：直接复用 `filterEngine.evaluateCondition`，只是把 `raw` 换成 "目标记录的 refFieldId 值"，把 `cond.value` 换成：
- `valueType=constant` → 原 value
- `valueType=field` → `record_i.cells[currentFieldId]`

这样运算符集合、空值语义、日期相对值处理等全部自动对齐。

### 4.3 返回值类型规则

| calcMethod | 返回类型 | 展示格式 |
|---|---|---|
| original, deduplicate | 数组，元素类型 = 目标字段原类型 | 按目标字段格式渲染，多值用分隔符 |
| count, deduplicateCount | number | 整数 |
| sum, average | number | `lookupOutputFormat` 或目标字段 numberFormat |
| max, min | number / date | 跟随目标字段类型 |

## 5. 读路径集成

### 5.1 `viewEngine.queryView` 改造

```
原流程: filter → sort → group → paginate
新流程: materializeLookups → filter → sort → group → paginate
```

`materializeLookups(records, table)`：
- 遍历 `table.fields`，对每个 `type === "Lookup"` 的字段
- 调 `computeLookupBatch` 得到 `Map<recordId, cellValue>`
- 把结果临时挂到每条 record 的 `cells[field.id]`（内存对象，不写 DB）

这样 filter/sort/group 对 Lookup 字段一视同仁。

### 5.2 筛选/排序/分组对 Lookup 的支持

- `filterEngine.ts:213` 当前占位逻辑删除；改为在 materialize 之后按 **目标字段的 underlying type** 走对应分支
  - 需要在 `evaluateCondition` 签名上允许传入 `effectiveType`
  - Lookup 的 effectiveType = `refField.type`；若 calcMethod ∈ {count, dedupeCount} 则强制为 Number
- sortEngine / groupEngine 同理

### 5.3 嵌套 Lookup

允许 `refField.type === "Lookup"`。
- 采用**按字段拓扑排序**解决：在 materialize 前构建 Lookup 依赖 DAG，按拓扑序依次 materialize
- 检测环路：给出 `LookupCircularError`，字段值显示 `#CYCLE!`

## 6. 写路径与校验

### 6.1 字段创建/编辑

新增 `validateLookupConfig(config, tables)`：
1. `refTableId` 必须存在（允许 = 当前表 id，支持自查询）
2. `refFieldId` 必须是 refTable 的字段，且不能是**当前正在编辑的这个 Lookup 字段**（防止自引用）
3. `conditions.length <= 5`
4. 每条 condition：
   - `refFieldId` 属于 refTable
   - `operator` 合法且与 `refFieldId.type` 兼容（用 `fieldValidator` 现有映射）
   - `valueType=field` 时 `currentFieldId` 必须属于当前表
5. 检测跨 Lookup 成环（扫描目标字段的 lookup 依赖图）

### 6.2 记录写入

`fieldValidator.ts` 现有 `Lookup → 只读` 逻辑保留不动。

### 6.3 字段删除/重命名的级联

- 目标表删除某字段 → 扫描所有 Lookup 字段，把 refFieldId / condition.refFieldId 命中者标记为 `broken: true`，前端显示 `#REF!`，允许用户修复
- 目标表被删除 → 同上
- 本表字段删除（被 condition 的 currentFieldId 引用）→ 同上

## 7. 更新与一致性

### 7.1 阶段一：按需计算（Phase 1，首版）

- 不做缓存，每次 `queryView` 都 materialize
- 优点：零一致性问题
- 代价：`O(N_current × N_ref)` per query；对小数据集（项目默认规模）可接受

### 7.2 阶段二：粒度缓存（可选）

- Record 新增 `derivedCells: Json`
- Record 写入时：若当前表对某 Lookup 字段是 **conditions 里被引用的 currentFieldId**，标记 dirty
- 目标表记录写入/删除时：标记所有"以此表为 refTable 的 Lookup"字段 dirty
- 下次 query 时对 dirty 字段重算并写回

### 7.3 实时推送（Phase 3，非必需）

通过已有的 SSE 通道把 Lookup 结果变更推给前端，和 AI filter 的流式响应共用基础设施。

## 8. API 变更

### 8.1 字段接口

`POST /api/tables/:id/fields`、`PATCH /api/fields/:id` 的 body 里 `config.lookup` 需要通过 `validateLookupConfig`。返回 400 时错误结构：

```json
{
  "error": "LOOKUP_CONFIG_INVALID",
  "details": { "field": "conditions[0].refFieldId", "reason": "field not found" }
}
```

### 8.2 视图查询

`GET /api/views/:id/records` 的响应结构不变；Lookup 的值直接出现在 `records[i].cells[lookupFieldId]` 中。前端无需新字段。

### 8.3 调试接口（可选）

`POST /api/fields/:id/lookup/preview`：给定一条 record（或 recordId），返回 Lookup 计算的中间结果（命中的 refRecords + 原始值数组 + 聚合结果），方便 UI 里"查看匹配详情"。

---

## 9. 前端改造

### 9.1 `frontend/src/types.ts`

把 Lookup 相关类型从 backend 同步过来（或改为共享 types 包）。当前前端 types 只覆盖 8 种字段，需要扩展。

### 9.2 字段配置面板

新增 `LookupFieldConfig.tsx`，在字段创建/编辑侧栏里当 `type === "Lookup"` 时展示：

```
┌─────────────────────────────────────────┐
│  查找引用配置                             │
├─────────────────────────────────────────┤
│  引用数据表   [ 任务分配表 ▼ ]            │
│  引用字段     [ 工时 ▼ ]                 │
│                                          │
│  查找条件    满足 [所有 ▼] 条件时命中      │
│  ┌────────────────────────────────────┐ │
│  │ 源表.负责人 [=▼] 当前表.负责人 [✕] │ │
│  │ 源表.状态  [=▼] "进行中"    [✕]   │ │
│  │ + 添加条件（最多 5 条）            │ │
│  └────────────────────────────────────┘ │
│                                          │
│  计算方式     [ 求和 ▼ ]                 │
│  显示格式     [ 数字 · 整数 ▼ ]          │
└─────────────────────────────────────────┘
```

交互要点：
- 「引用字段」下拉按 refTable 的 fields 动态刷新
- 每个条件行的左侧字段选择器限定为 refTable 字段；右侧值支持 `字段 | 常量` 切换
- 计算方式可选项随 refField.type 过滤（字符串字段不展示 sum/avg）

### 9.3 单元格渲染

`TableView` 对 Lookup 单元格：
- 只读样式（禁用双击编辑）
- 按 calcMethod + effectiveType 走对应渲染器（复用现有的单选/日期/数字渲染器）
- 若为数组（original/dedupe）用 chip 列表展示

## 10. 边界与异常

| 情况 | 行为 |
|---|---|
| refTable 不存在 / 被删 | 单元格显示 `#REF!`，字段标 `broken` |
| refField 不存在 / 被删 | 同上 |
| 条件中引用当前表字段被删 | 同上（只坏这一条 condition）|
| 命中 0 条 | count/dedupeCount → `0`；sum → `0`；其他 → `null` |
| 聚合类型不匹配（对文本 sum）| 跳过不可数值项；若全部跳过返回 `null` |
| 目标表自身是当前表（自查询）| 允许；但 refField 不能是正在编辑的这个 Lookup 字段 |
| Lookup 嵌套成环 | 抛 `LookupCircularError`，显示 `#CYCLE!` |
| condition 中对 Lookup/Formula 字段做匹配 | 允许；在 materialize 拓扑序下已保证其值可用 |

## 11. 实施计划

### Phase 1（MVP，约 3~4 个 PR）
1. **P1-1 数据模型补丁**：`LookupCondition.currentFieldId`、config 校验器、前端 types 同步
2. **P1-2 lookupEngine**：`computeLookup` + `computeLookupBatch`，单测覆盖 8 种 calcMethod
3. **P1-3 viewEngine 集成**：materialize 流程 + 拓扑排序 + 环检测
4. **P1-4 前端配置面板 + 单元格渲染**

Phase 1 交付后，Lookup 即可作为只读派生字段在表格中显示、被筛选/排序/分组。

### Phase 2
- 跨字段变更级联标记 broken
- 调试 preview 接口
- `derivedCells` 粒度缓存（若 Phase 1 性能不够）

### Phase 3（可选）
- SSE 实时刷新
- 与 AI Filter 联动：让自然语言可以生成"创建一个查找引用字段"的指令

---

## 12. 风险与取舍

| 风险 | 缓解 |
|---|---|
| 每次查询都 materialize → 大表性能 | Phase 1 先接受，Phase 2 做 dirty 缓存；当前项目数据集小 |
| 嵌套 Lookup 复杂度 | 拓扑排序 + 环检测；限制最大嵌套深度 = 5 |
| 条件运算符与 refField 类型不兼容 | 在字段保存时校验，给出明确错误而非运行时报错 |
| 前后端 types 漂移 | 建议抽出 `shared/types.ts` 包（非本期必做）|
| 用户误配置成笛卡尔式匹配（无条件）| UI 在条件列表为空时 warning，保存时允许但提示 |

## 13. 示意图

### 13.1 数据流

```
  ┌───────────────┐              ┌──────────────────┐
  │ 当前表 records │ ─┐           │  目标表 records   │
  └───────────────┘  │           └──────────────────┘
                     │                    │
                     ▼                    ▼
            ┌────────────────────────────────────┐
            │         lookupEngine               │
            │  ① 按 conditions 过滤目标表         │
            │  ② 取 refFieldId 的值数组           │
            │  ③ 按 calcMethod 聚合               │
            └──────────────┬─────────────────────┘
                           ▼
         record.cells[lookupFieldId] = <派生值>（内存临时挂载）
                           ▼
             filter / sort / group / 渲染
```

### 13.2 配置例（任务跟进表 · 本周总工时）

```
refTableId: "tbl_task_assign"
refFieldId:  "fld_hours"
conditionLogic: "and"
conditions: [
  { refFieldId: "fld_owner",  operator: "eq",     valueType: "field",    currentFieldId: "fld_owner" },
  { refFieldId: "fld_status", operator: "eq",     valueType: "constant", value: "进行中" },
  { refFieldId: "fld_due",    operator: "before", valueType: "constant", value: "next7Days" }
]
calcMethod: "sum"
```

运行时效果：对当前表里每个"负责人"，把任务分配表中 `负责人相同 AND 状态=进行中 AND 截止早于未来 7 天` 的所有任务工时求和。
